# Documentation maintenance for agents

Docs in this repo are **segmented**. Each file owns a narrow scope. When you
make a code change, update **only** the doc segments that own the thing you
changed — don't rewrite anything else.

## The map

| When you change… | Update this doc |
| --- | --- |
| Adding/removing a Conventional Commits scope, release tooling, versioning | [`/CLAUDE.md`](../CLAUDE.md) → *Commit messages & releases* |
| Architecture, conventions, tech stack, env vars, dev commands | [`/CLAUDE.md`](../CLAUDE.md) → relevant section |
| Public-facing project info (what it is, how to run it, links) | [`/README.md`](../README.md) |
| Anything inside a feature module (routes, actions, services, schemas, tables) | [`docs/modules/<module>.md`](./modules/) |
| Typography / font / type system | [`docs/TYPOGRAPHY.md`](./TYPOGRAPHY.md) |
| Server-action patterns (the *how*, not a specific action) | [`/app/actions/AGENTS.md`](../app/actions/AGENTS.md) |
| Service-layer patterns (the *how*, not a specific service) | [`/lib/services/AGENTS.md`](../lib/services/AGENTS.md) |
| Agent skills / reusable playbooks | [`/.github/skills/`](../.github/skills/) |

If a change touches **two segments** (e.g. a new module *and* a new convention),
update **both** — but each one stays scoped to its own concern. No copy-paste
between docs.

## Module docs — fixed shape

Every file in [`docs/modules/`](./modules/) follows
[`_TEMPLATE.md`](./modules/_TEMPLATE.md) exactly:

```
Status / Last reviewed / Overview / Routes / Server actions / Services /
Schemas / Types / Components / DB tables / Notes
```

Rules:

1. **Same sections, same order, every file.** Don't add new section headings —
   put notes under *Notes*.
2. **Bump "Last reviewed" to today** whenever you touch the module's code or its
   doc. Use `YYYY-MM-DD`.
3. **Touch the line, not the doc.** If you add a route, add one bullet under
   *Routes*. Don't reflow the whole file.
4. **One-line entries.** Each bullet stays a one-line summary. If you need a
   paragraph, the *Notes* section is the only place for it.
5. **No invention.** If a section has nothing yet (e.g. no schemas), write
   "—". Don't fabricate file names.
6. **New module → copy `_TEMPLATE.md`.** Register it in
   [`docs/modules/README.md`](./modules/README.md) **and** in this map above.

## What to do on every meaningful change

Run this checklist mentally before closing the task:

- [ ] Did I add/remove/rename a route, action, service, schema, type, or DB
      table? → update the module doc's bullet.
- [ ] Did I add a new module? → copy `_TEMPLATE.md`, register in
      `docs/modules/README.md` and the map above.
- [ ] Did I add a new Conventional Commits scope?
      → add it to [`commitlint.config.mjs`](../commitlint.config.mjs) **and**
      the relevant module doc's *Notes*.
- [ ] Did I change an env var, command, or required tool? → update
      [`CLAUDE.md`](../CLAUDE.md) and (if user-facing) [`README.md`](../README.md).
- [ ] Did I touch a public-facing surface (landing, public trip share, signup)?
      → update [`README.md`](../README.md).

## What NOT to do

- ❌ Don't write a "changelog" inside a doc — that's what `CHANGELOG.md` is for
  (generated by release-please).
- ❌ Don't duplicate content across docs. Link instead.
- ❌ Don't create new top-level doc files without a clear scope that none of the
  existing ones cover. Prefer extending an existing segment.
- ❌ Don't rewrite a doc just because it feels stale — make a targeted edit to
  the line that's wrong.
- ❌ Don't add long prose explanations. If you can't say it in a line, it
  belongs in code comments or a focused doc under `docs/`.

## When in doubt

Ask: *which single sentence in which single file is wrong because of my change?*
Edit that. Stop.
